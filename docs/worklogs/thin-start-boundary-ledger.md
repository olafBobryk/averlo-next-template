# Thin-Start Boundary Ledger

## Purpose

- Goal: Consolidate the optional thin-start creation boundary into verified wording and, only after user approval, the right durable project doc or ADR.
- Product spine: Make the optional thin-start creation boundary unambiguous without re-deciding settled points or implementing the creation path.
- Success criteria: Existing decisions are mapped to repo docs or the verified chat packet, gaps are labeled as documentation gaps rather than decision gaps, wording is approved in chat before any ADR/doc change, and a fresh-chat handoff can resume deterministically.
- Ledger owner: Codex agent with user verification at human gates.
- Last updated: 2026-06-09

## Operating Rules

- Default autonomy: Proceed inside evidence gathering, source mapping, and draft preparation; stop before final wording is treated as accepted or written to durable docs.
- Human gates: Any final ADR/doc wording, any change to the thin-start component boundary, any implementation/backport/prune work, and any decision that would reopen a settled point.
- Decision/discussion flow: Discuss wording in chat first. Record only accepted outcomes here.
- Verification baseline: Local doc/source search, Template Intelligence topics where available, and explicit user approval for decision wording.
- Handoff file: `docs/worklogs/thin-start-boundary-handoff.md`
- Rolling window size: Current chunk plus next 2 to 3 meaningful chunks.

## Status Legend

- `queued`: not started
- `in_progress`: actively being worked
- `blocked`: cannot proceed without external fix or missing dependency
- `needs_human`: decision gate reached
- `fixed`: implementation or review pass complete, not yet verified
- `verified`: checks passed for the chunk
- `signed_off`: human accepted the chunk
